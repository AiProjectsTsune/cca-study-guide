超初心者向け - まずこの用語を理解しよう

試験範囲に出てくる専門用語を、日常の例え話で解説します。

API（エーピーアイ）
レストランの注文窓口のようなもの。お客さん（あなたのプログラム）がメニュー（API仕様）を見て注文（リクエスト）すると、キッチン（サーバー）が料理（データ）を返してくれます。Claude APIは「Claudeに質問を送って回答をもらう窓口」です。
エージェント（Agent）
自分で考えて行動する秘書のようなもの。普通のチャットボットは「聞かれたことに答えるだけ」ですが、エージェントは「ファイルを読む」「検索する」「コードを実行する」など、自分でツールを使って仕事を完了してくれます。
MCP（Model Context Protocol）
USBポートのようなもの。USBポートがあれば、マウスもキーボードもプリンターも繋げますよね。MCPはClaude AIに「データベース」「ファイル」「Webサービス」などを統一規格で繋げる仕組みです。
プロンプトエンジニアリング
AIへの「上手な指示の出し方」。同じ質問でも、聞き方を工夫するとAIの回答品質が劇的に変わります。料理を注文するとき「美味しいもの」と言うより「辛さ控えめの鶏肉料理、ご飯付き」と言うほうが希望通りのものが来るのと同じです。
JSONスキーマ
注文用紙のフォーマット。「名前はここ」「住所はここ」「電話番号はここ」とデータの形を事前に決めておく仕組み。AIに「この形式で答えてね」と伝えることで、プログラムが処理しやすい回答を得られます。
コンテキストウィンドウ
AIの「短期記憶」の大きさ。会話が長くなると古い内容を忘れてしまいます。このメモリの容量がコンテキストウィンドウで、Claudeは最大200Kトークン（本数冊分）まで覚えられます。
CLAUDE.md
Claudeへの「取扱説明書」。プロジェクトのルート（一番上のフォルダ）に置くファイルで、「このプロジェクトではTypeScriptを使って」「テストは必ずJestで」などの指示を書いておくと、Claude Codeが毎回それに従って動きます。
オーケストレーション
指揮者のようなもの。複数のエージェントを「あなたはリサーチ担当」「あなたはコーディング担当」と役割分担させて、全体をまとめ上げること。一人でやるより効率的に大きな仕事をこなせます。
SDK（Software Development Kit）
工具セットのようなもの。家具を組み立てるのに必要な工具（ドライバー、レンチ等）がまとめて入っている箱。Agent SDKは「エージェントを作るために必要なコードやツールが全部入ったパッケージ」です。
CI/CD
自動出版システム。原稿（コード）を書いたら、自動で校正（テスト）して、自動で印刷・配本（デプロイ）してくれる仕組み。Continuous Integration / Continuous Deliveryの略。

D1: エージェンティック・アーキテクチャ（27%）

最も配点が高い！ここを落とすと合格できない

エージェントの基本フロー: ユーザーリクエスト → エージェントループ → ツール実行 → レスポンス

超初心者向け解説: エージェントとは？

日常の例で理解する

普通のチャットAI = 質問に答えるだけの受付係

エージェント = 自分で電話して、書類を調べて、報告書を作ってくれる秘書

エージェントの動き方（エージェンティック・ループ）:

  1. 考える: 「この質問に答えるには何が必要かな？」
  2. 道具を選ぶ: 「データベースを検索しよう」
  3. 道具を使う: データベースに問い合わせる
  4. 結果を確認: 「十分な情報が集まったかな？」
  5. 繰り返す or 回答: 足りなければ1に戻る、十分なら回答する

出題されるトピック

Agent SDK

Anthropicが提供するエージェント開発キット。エージェントの作成・実行・管理ができる。

マルチエージェント

複数のエージェントを連携させる設計。「調査エージェント」「執筆エージェント」など役割分担。

フック（Hooks）

エージェントの動作の前後に独自の処理を挟む仕組み。「ツール実行前にログを残す」など。

タスク分解

大きな仕事を小さなステップに分割するパターン。複雑な問題を解きやすくする。

コード例: 基本的なエージェント

import anthropic

# エージェントのツールを定義
tools = [
    {
        "name": "search_database",
        "description": "商品データベースを検索する",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {
                    "type": "string",
                    "description": "検索キーワード"
                }
            },
            "required": ["query"]
        }
    }
]

# エージェンティック・ループ
client = anthropic.Anthropic()
messages = [{"role": "user", "content": "赤いスニーカーを探して"}]

while True:
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        tools=tools,
        messages=messages
    )

    # ツール使用が必要なら実行して結果を返す
    if response.stop_reason == "tool_use":
        # ツールを実行して結果をmessagesに追加
        # ... ループ継続
        pass
    else:
        # 最終回答 → ループ終了
        print(response.content[0].text)
        break

D2: ツール設計 & MCP統合（18%）

超初心者向け解説: MCPとは？

日常の例で理解する

スマートフォンを考えてみてください:

  • スマホ本体 = Claude AI
  • アプリ = MCP サーバー（データベース、メール、カレンダーなど）
  • App Store = MCPの統一規格（どんなアプリも同じルールで動く）

MCPがあることで、Claude はいろんな外部サービスに統一的な方法で接続できます。

出題されるトピック

  • ツール説明のベストプラクティス: ツール名と説明を明確に書く（AIはこれを読んでツールを選ぶ）
  • 構造化エラー対応: ツールがエラーを返す場合のハンドリング
  • MCP設定: claude_desktop_config.json でサーバーを接続
  • Claudeの組み込みツール: Read, Write, Edit, Bash, Glob, Grep
  • ツールの粒度設計: 1つのツールに機能を詰め込みすぎない
よくある間違い（アンチパターン）
  • ツールの説明が曖昧 → AIが適切なツールを選べない
  • 1つのツールに複数の機能 → 使い分けが困難に
  • エラー時の処理がない → エージェントが止まってしまう

超初心者向け解説: Claude Codeとは？

日常の例で理解する

Claude Code = あなたのPCの中で動くAIプログラマー助手

ターミナルで claude と打つだけで起動。ファイルの読み書き、コード修正、テスト実行、Git操作まで全部やってくれます。

普通のChatGPTやClaude.aiとの違い:

トピック1: CLAUDE.md 設定（超重要）

一言でいうと

CLAUDE.md = Claudeへの「仕事のルールブック」

プロジェクトに置いておくと、Claude Codeがセッション開始時に毎回自動で読み込むファイルです。

CLAUDE.md の階層構造（公式ドキュメント準拠）

CLAUDE.mdは3つのレベルで設定でき、それぞれ影響範囲が異なります。

📁 パソコン全体（グローバル設定）
├── ~/.claude/CLAUDE.md              ← 全プロジェクト共通のルール
│
📁 プロジェクト（プロジェクト設定）
├── my-project/CLAUDE.md             ← このプロジェクト専用のルール
├── my-project/.claude/rules/*.md    ← ルールファイル（複数ファイルに分割可能）
│
📁 サブフォルダ（フォルダ設定）
├── my-project/src/CLAUDE.md         ← src/ で作業するときだけ有効
├── my-project/tests/CLAUDE.md       ← tests/ で作業するときだけ有効
試験で問われるポイント
  • 読み込み順序: グローバル → プロジェクト → フォルダ の順で読み込まれる
  • 優先順位: フォルダ > プロジェクト > グローバル（より具体的な設定が勝つ）
  • .claude/rules/: ルールをファイルごとに分割して管理できる。paths: フロントマターで特定のファイルにだけ適用することも可能
  • 除外設定: claudeMdExcludes で不要なCLAUDE.mdを無視できる（モノレポで重要）

CLAUDE.md の書き方 - 実践例

# プロジェクトルール（CLAUDE.md の例）

## 技術スタック
- 言語: TypeScript (strict mode)
- フレームワーク: Next.js 15 (App Router)
- スタイリング: Tailwind CSS
- テスト: Vitest + React Testing Library

## コーディング規約
- 変数名はキャメルケース（例: userName）
- コンポーネントはPascalCase（例: UserProfile）
- 関数は必ず型注釈をつける
- console.logは本番コードに残さない

## ビルドコマンド
- 開発: npm run dev
- テスト: npm test
- ビルド: npm run build

## 禁止事項
- any型の使用禁止
- .envファイルをコミットしない
- 外部APIキーをハードコードしない

このように書いておくと、Claude Codeは毎回このルールに従ってコードを書いてくれます。

パスごとのルール適用（.claude/rules/）

公式ドキュメントでは、.claude/rules/ディレクトリにYAMLフロントマター付きのMDファイルを置いて、特定のファイルにだけルールを適用できます。

# .claude/rules/api-rules.md

---
paths:
  - "src/api/**/*.ts"     ← src/api/ 内のTypeScriptファイルにだけ適用
---

APIエンドポイントでは必ず以下を守ること:
- 入力バリデーションにZodを使用
- エラーレスポンスは統一フォーマットで返す
- レートリミットを設定する

自動メモリ（Auto Memory）

自動メモリとは？
Claude Codeが作業中に自動で学習したこと（ビルドコマンド、デバッグの知見など）を次のセッションでも覚えている仕組み。あなたが何も書かなくても、Claudeが勝手に学習メモを保存してくれます。

トピック2: カスタムスラッシュコマンド

一言でいうと

よく使う作業を「/コマンド名」で一発実行できる仕組み

料理のレシピカードのようなもの。「/カレー」と言えば、材料の準備から盛り付けまでの手順が自動で実行される。

カスタムコマンドの作り方

.claude/commands/フォルダにマークダウンファイルを置くだけ！

📁 プロジェクト/
├── .claude/
│   └── commands/
│       ├── review.md        ← /review で実行される
│       ├── deploy.md        ← /deploy で実行される
│       ├── ci/
│       │   ├── build.md     ← /build で実行される
│       │   └── test.md      ← /test で実行される
│       └── git/
│           ├── commit.md    ← /commit で実行される
│           └── pr.md        ← /pr で実行される

コマンドファイルの書き方（YAMLフロントマター）

ファイルの先頭にYAMLフロントマター（設定情報）を書けます。

# .claude/commands/review.md

---
description: "コードレビューを実行する"     ← コマンドの説明
allowed-tools: "Read, Bash(git:*)"          ← 使えるツールを制限
model: "sonnet"                              ← 使用するモデルを指定
argument-hint: "[ファイルパス]"              ← 引数のヒント
---

以下の手順でコードレビューを行ってください:

1. 指定されたファイルを読み込む
2. バグ、セキュリティの問題、パフォーマンスの問題をチェック
3. 改善提案をリストアップ
4. 重要度順に並べて報告

レビュー基準:
- SOLID原則に従っているか
- エラーハンドリングが適切か
- テストが書かれているか
YAMLフロントマターの主要フィールド（試験出題範囲）
フィールド意味例
descriptionコマンドの説明文"PRを作成する"
allowed-tools使用可能なツールの制限"Read, Bash(git:*)"
model使用モデルの指定"sonnet" / "opus"
argument-hint引数のヒント表示"[PR番号] [優先度]"
disable-model-invocationAI呼び出しなしで実行true

スキル（SKILL.md）- 自動発動するコマンド

通常のコマンドは /コマンド名 で手動実行しますが、スキル（SKILL.md）は条件に合致すると自動的に発動します。

# .claude/commands/explain-code/SKILL.md

---
name: explain-code
description: コードの仕組みを図解付きで説明する。
  「これどう動くの？」「仕組みを教えて」と聞かれたら自動発動。
---

コードを説明するときは必ず以下を含めること:

1. 日常の例えで全体像を説明
2. ASCII図で構造を描く
3. ステップバイステップでコードを解説
4. よくある間違いを指摘

トピック3: プランモード

一言でいうと

「いきなり作業せず、まず計画書を見せて」モード

家のリフォームで例えると: いきなり壁を壊し始めるのではなく、まず設計図を見せてもらって「OK」を出してから作業開始する。

プランモードの使い方

# 方法1: キーボードショートカット
Shift+Tab       ← プランモードのON/OFFを切り替え

# 方法2: セッション開始時に指定

トピック4: フック（Hooks）

一言でいうと

Claudeの動作の前後に「自動チェック」を挟む仕組み

工場のベルトコンベアで例えると: 製品（コード）が次の工程に進む前に、自動で品質検査を入れるイメージ。

フックの種類（公式ドキュメント準拠）

フック名発動タイミング使用例
PreToolUse ツール使用の前 Bashコマンド実行前に危険なコマンドをブロック
PostToolUse ツール使用の後 ファイル編集後に自動フォーマット・Lint実行

フックの設定例（settings.json）

// .claude/settings.json に記述

{
  "hooks": {

    // ファイル編集後に自動でコードスタイルをチェック
    "PostToolUse": [
      {
        "matcher": "Write|Edit",       // Write か Edit ツール使用後に発動
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/check-style.sh"  // スタイルチェック実行
          }
        ]
      }
    ],

    // Bashコマンド実行後にログを記録
    "PostToolUse": [
      {
        "matcher": "Bash",             // Bashツール使用後に発動
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.command' >> ~/.claude/command-log.txt"
          }
        ]
      }
    ]
  }
}
試験で問われるポイント
  • matcher: どのツールに対してフックを発動するか（"Bash", "Write|Edit", "*" など）
  • type: "command"（シェルコマンド実行）が基本
  • $CLAUDE_PROJECT_DIR: プロジェクトルートのパスを参照する環境変数
  • timeout: フックのタイムアウト（デフォルト60秒）

トピック5: CI/CD統合（非対話モード）

一言でいうと

Claude を「自動化パイプライン」の一部として使う方法

普段は対話（チャット）で使いますが、CI/CDでは人間なしで自動実行する必要があります。そのための「ヘッドレス（画面なし）モード」です。

非対話モードの基本（-p フラグ）

# -p フラグ = 「プロンプトを渡して即実行、結果を出力して終了」

# テキスト出力（人間が読む用）
claude -p "このコードのバグを見つけて" --output-format text

# JSON出力（プログラムが処理する用）
claude -p "このコードを分析して" --output-format json

# ストリームJSON（リアルタイム処理用）
claude -p "ログを解析して" --output-format stream-json

パイプ処理（他のコマンドと連携）

Linuxのパイプ（|）を使って、他のツールと連携できます。

# ファイルの中身をClaudeに渡して分析結果をファイルに保存
cat data.txt | claude -p "このデータを要約して" --output-format text > summary.txt

# コードを渡してバグ分析結果をJSONで取得
cat code.py | claude -p "バグを分析して" --output-format json > analysis.json

# ログファイルからエラーを抽出
cat error.log | claude -p "エラーの原因を分析して" --output-format json | jq '.result'

GitHub Actions での使い方

トピック3: tool_use による構造化出力の強制

一言でいうと

AIに「この形で答えて」と"強制"する仕組み

普通にAIに聞くと自由な文章で返ってきますが、プログラムで処理するには決まった形式のデータが必要です。構造化出力はそれを保証します。

日常の例で理解する

構造化出力 = 注文用紙への記入

構造化出力なし: 「鈴木さんがメールでエンタープライズプランに興味があると言ってました」 → プログラムで処理しにくい

構造化出力あり:

{
  "name": "鈴木太郎",
  "email": "suzuki@example.com",
  "plan_interest": "enterprise",
  "demo_requested": true
}

→ プログラムがすぐにデータベースに保存できる！

公式の2つの方法（試験必出）

Anthropic公式ドキュメントでは、構造化出力を実現する2つの方法が提供されています。

方法仕組み保証レベル用途
JSON出力
(output_config)		 レスポンス全体をJSONスキーマに従わせる スキーマ準拠を保証 データ抽出、レポート生成、分類
Strict Tool Use
(strict: true)		 ツール呼び出しの引数をスキーマに従わせる 型安全を保証 エージェントのツール呼び出し、関数実行

方法1: JSON出力（output_config.format）

AIの回答全体を指定したJSONスキーマの形に強制します。

# Python（Pydanticモデルを使う方法 - 公式推奨）
from pydantic import BaseModel
from anthropic import Anthropic

# ステップ1: 「注文用紙」の型を定義する
class ContactInfo(BaseModel):
    name: str               # 名前（文字列）
    email: str              # メール（文字列）
    plan_interest: str      # 興味のあるプラン（文字列）
    demo_requested: bool    # デモ希望かどうか（true/false）

client = Anthropic()

# ステップ2: output_format にPydanticモデルを渡す
response = client.messages.parse(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{
        "role": "user",
        "content": "抽出して: 鈴木太郎 (suzuki@example.com) "
                   "エンタープライズプランに興味あり、火曜14時にデモ希望"
    }],
    output_format=ContactInfo   # ← ここで型を指定！
)

# ステップ3: 自動でバリデーション済みのデータが得られる
print(response.parsed_output)
# → ContactInfo(name='鈴木太郎', email='suzuki@example.com',
#               plan_interest='enterprise', demo_requested=True)
SDK別の書き方（試験に出る可能性あり）
言語メソッドスキーマ定義方法
Pythonclient.messages.parse()Pydantic BaseModel
TypeScriptclient.messages.parse()Zod スキーマ + zodOutputFormat()
cURL / その他output_config パラメータ生の JSON Schema

方法2: Strict Tool Use（strict: true）

ツール呼び出しの引数がスキーマに100%準拠することを保証します。

# エージェントのツールに strict: True を設定
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "東京の天気を教えて"}],
    tools=[{
        "name": "get_weather",
        "description": "指定した都市の現在の天気を取得する",
        "strict": True,     # ← これが重要！スキーマ準拠を保証
        "input_schema": {
            "type": "object",
            "properties": {
                "location": {
                    "type": "string",
                    "description": "都市名（例: 東京）"
                },
                "unit": {
                    "type": "string",
                    "enum": ["celsius", "fahrenheit"]  # この2つしか選べない
                }
            },
            "required": ["location"],       # location は必須
            "additionalProperties": False   # 余計なフィールドを禁止
        }
    }]
)

# strict: True の保証:
# ✅ JSONが壊れることはない
# ✅ required フィールドが必ず存在する
# ✅ 型が正しい（stringにnumberが入らない）
# ✅ enum以外の値が入らない
# ✅ 再試行ループが不要になる！
試験で問われる JSON Schema の制約（公式ドキュメント）

使えるキーワード:

  • type, properties, required, additionalProperties
  • enum（選択肢を限定）, const（固定値）
  • items（配列の要素定義）, description

使えないキーワード（要注意！）:

  • minimum, maximum（数値の最小/最大） → descriptionに記述で代用
  • minLength, maxLength（文字列長） → descriptionに記述で代用
  • pattern（正規表現） → 非対応

SDKが自動でこれらの制約を description に変換してくれます。

Claude Code CLI での構造化出力

Claude Code の CLI でも --json-schema フラグで構造化出力が使えます。

# CLIで構造化出力を取得
claude -p "auth.py の関数名を抽出して" \
  --output-format json \
  --json-schema '{
    "type": "object",
    "properties": {
      "functions": {
        "type": "array",
        "items": {"type": "string"}
      }
    },
    "required": ["functions"]
  }'

# 結果例:
# {"functions": ["login", "verify_token", "logout", "refresh_session"]}

ベストプラクティス（公式推奨）

DO（やるべき）

  • Pydantic / Zod でスキーマ定義（生JSONより安全）
  • additionalProperties: false を必ず設定
  • description を各フィールドに付ける
  • 選択肢は enum で制限する

DON'T（やってはいけない）

  • プロンプトだけでJSON出力を「お願い」する（保証されない）
  • minimum/maximum をスキーマに入れる（無視される）
  • 巨大すぎるスキーマを定義する（性能低下）
  • strict: true なのに再試行ループを書く（無駄）

トピック4: 検証ループと再試行ループ

一言でいうと

AIの回答を「検品」して、不良品なら「やり直し」させる品質管理システム

工場の品質管理で例えると: 製品を検査して、不良品があれば製造ラインに戻してやり直させる。それと全く同じです。

重要: structured outputs との関係

公式ドキュメントでは、structured outputs（strict: true）を使えば検証ループは不要になると明記されています。

検証ループが必要なのは:

  • strict: true を使わない場合
  • スキーマの制約（minimum/maximum等）を自分で検証する場合
  • ビジネスロジック上の妥当性を検証する場合（例: 日付が未来であるか）

検証ループの全体フロー

フロー図（日常の例え付き） 
┌──────────────────────────────────────────┐
│  1. AIにリクエストを送る                    │ ← 「この注文を処理して」
│     (プロンプト + スキーマ指定)              │
└─────────────┬────────────────────────────┘
              ▼
┌──────────────────────────────────────────┐
│  2. AIがレスポンスを返す                    │ ← 料理が出てくる
│     (JSON形式のデータ)                      │
└─────────────┬────────────────────────────┘
              ▼
┌──────────────────────────────────────────┐
│  3. 検証（バリデーション）                   │ ← 品質検査
│     ├── JSONとして正しいか？                │
│     ├── 必須フィールドがあるか？             │
│     ├── データ型は正しいか？                │
│     └── ビジネスルールを満たすか？           │
└─────┬───────────────┬────────────────────┘
      │ ✅ 合格        │ ❌ 不合格
      ▼               ▼
┌──────────┐  ┌───────────────────────────┐
│ 結果を返す │  │ 4. エラー内容をフィードバック  │
│ （完了）   │  │    して再度リクエスト          │
└──────────┘  │    （最大N回まで）              │
              └──────────┬────────────────┘
                         │ 再試行 (→ ステップ2に戻る)
                         ▼
              ┌───────────────────────────┐
              │ 5. 最大回数に達したら       │
              │    エラーとして処理         │
              └───────────────────────────┘

実装例: 完全な検証ループ（Python）

import anthropic
import json

client = anthropic.Anthropic()

def extract_with_validation(text: str, max_retries: int = 3) -> dict:
    """検証ループ付きのデータ抽出"""

    # 期待するスキーマの定義
    required_fields = ["name", "email", "age"]

    messages = [{
        "role": "user",
        "content": f"""以下のテキストから情報を抽出して、
JSONで返してください。
必須フィールド: name(文字列), email(文字列), age(数値)

テキスト: {text}"""
    }]

    for attempt in range(max_retries):
        # ---- ステップ1: AIにリクエスト ----
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            messages=messages
        )
        raw_output = response.content[0].text

        # ---- ステップ2: JSON パース検証 ----
        try:
            data = json.loads(raw_output)
        except json.JSONDecodeError as e:
            # JSONが壊れている → フィードバックして再試行
            messages.append({"role": "assistant", "content": raw_output})
            messages.append({
                "role": "user",
                "content": f"JSONパースエラーです: {e}\n"
                           f"正しいJSONで返してください。"
            })
            continue  # → ループの先頭に戻る

        # ---- ステップ3: フィールド存在チェック ----
        missing = [f for f in required_fields if f not in data]
        if missing:
            messages.append({"role": "assistant", "content": raw_output})
            messages.append({
                "role": "user",
                "content": f"以下のフィールドが不足: {missing}\n"
                           f"すべて含めて再出力してください。"
            })
            continue  # → ループの先頭に戻る

        # ---- ステップ4: 型チェック ----
        if not isinstance(data.get("age"), (int, float)):
            messages.append({"role": "assistant", "content": raw_output})
            messages.append({
                "role": "user",
                "content": "age は数値で返してください。"
            })
            continue  # → ループの先頭に戻る

        # ---- 全検証クリア！ ----
        return data

    # 最大回数に達した場合
    raise ValueError(f"{max_retries}回の試行で有効なデータを取得できませんでした")

# 使用例
result = extract_with_validation("鈴木太郎、35歳、suzuki@example.com")
print(result)
# → {"name": "鈴木太郎", "email": "suzuki@example.com", "age": 35}

structured outputs で検証ループが不要になる例

上記のコードは strict: true や output_format を使えば大幅に簡略化できます。

from pydantic import BaseModel
from anthropic import Anthropic

# スキーマを定義するだけ
class PersonInfo(BaseModel):
    name: str
    email: str
    age: int

client = Anthropic()

# これだけでOK！検証ループ不要！
response = client.messages.parse(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{
        "role": "user",
        "content": "抽出: 鈴木太郎、35歳、suzuki@example.com"
    }],
    output_format=PersonInfo  # ← スキーマ準拠が保証される
)

print(response.parsed_output)
# → PersonInfo(name='鈴木太郎', email='suzuki@example.com', age=35)
# JSONパースエラー → 起きない
# フィールド不足 → 起きない
# 型エラー → 起きない

それでも検証ループが必要なケース

structured outputs でカバーできない検証
  • ビジネスロジック検証: 「年齢が0〜150の範囲内か」「メールアドレスが正しい形式か」→ JSON Schema の minimum / pattern は非対応
  • 外部データとの整合性: 「この商品IDがデータベースに存在するか」
  • 複数フィールド間の整合性: 「開始日が終了日より前か」

これらの場合は、structured outputs で形式を保証しつつ、ビジネスロジックだけ検証ループで追加チェックするのがベストプラクティスです。

出力一貫性を高めるテクニック（公式ドキュメント）

テンプレート指定

出力の形を事前に見せる。AIは見本に合わせて回答する。JSON / XML / カスタム形式に対応。

例で制約する

Few-shotで「こう来たらこう返す」の例を見せる。抽象的な指示より例のほうがAIの理解度が高い。

検索で一貫性を保つ

ナレッジベースから情報を取得して、それを元に回答させる。AIの「知識」ではなく「事実」に基づく回答になる。

タスクの分割

複雑なタスクを小さなサブタスクに分割。各ステップでAIの注意力が集中し、一貫性が向上する。

D4 まとめ: 試験対策チェックリスト

これが答えられれば合格ライン
  • structured outputs の2つの方法（output_config と strict tool use）の違いを説明できる
  • Pydantic / Zod を使ったスキーマ定義の利点を知っている
  • additionalProperties: false を設定すべき理由を説明できる
  • JSON Schema で使えるキーワード / 使えないキーワードを区別できる
  • 検証ループのフロー（リクエスト→検証→フィードバック→再試行）を実装できる
  • strict: true を使えば検証ループが不要になる理由を説明できる
  • それでも検証ループが必要なケース（ビジネスロジック検証）を挙げられる
  • Few-shotプロンプティングで出力一貫性を高める方法を知っている

D5: コンテキスト管理 & 信頼性（15%）

超初心者向け解説

AIの「記憶力」をどう管理するか

AIとの会話が長くなると、最初の方の内容を忘れてしまいます。

これはコンテキストウィンドウ（AIの短期記憶）に限りがあるため。

この問題にどう対処するかが「コンテキスト管理」です。

出題されるトピック

1. 段階的要約のリスク

長い会話を「要約」して圧縮する手法。ただし要約の過程で重要な情報が抜け落ちるリスクがある。

2. コンテキスト配置

重要な情報を会話のどこに置くかで、AIの精度が変わる。最初と最後に重要情報を置くのが効果的。

3. エスカレーションパターン

試験頻出

AIが「自分には判断できない」と認識したとき、人間に判断を委ねる仕組み。

例: 返金額が$100を超える場合 → マネージャーに転送

4. マルチエージェントのエラー伝播

エージェントAがエラーを出したとき、エージェントBにどう伝えるか。エラーが連鎖して全体が止まらないようにする設計。

5. 信頼度較正（キャリブレーション）

AIの「確信度」を数値化して、確信度が低い回答は人間に確認する仕組み。「90%確信あり」vs「30%確信あり」で対応を変える。

合格のコツ

最重要アドバイス

「暗記」ではなく「設計判断力」が問われる試験です

すべての問題はシナリオベース。「この状況で最適な設計は何か？」を判断する力が必要です。

1. 公式ドキュメントを読む

Anthropicの公式ドキュメント（docs.anthropic.com）が最も重要な教材です。特にAgent SDK、MCP、Claude Codeのセクション。

2. 手を動かす

読むだけでは不十分。実際にエージェントを作り、MCPサーバーを構築し、CLAUDE.mdを設定する経験が必要。

3. アンチパターンを覚える

「何をすべきか」だけでなく「何をしてはいけないか」も出題される。18個のアンチパターンを把握する。

4. トレードオフを考える

「レイテンシ vs コスト」「安全性 vs スピード」など、正解が1つではない設計判断の練習をする。
